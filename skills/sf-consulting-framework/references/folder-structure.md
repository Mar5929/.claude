# SF Consulting Framework — Folder Structure

The structure must be:
- **Consistent across projects** — same layout every time, so Claude Code always knows where things are
- **Self-documenting** — browsable in an IDE without needing Claude to interpret it
- **Flexible** — accommodates simple epics (no features) and complex ones (with features) without awkward empty directories

---

## File Descriptions & Templates

### Project Root

#### `CLAUDE.md`
Quick-context entry point for Claude Code. Provides lightweight project orientation and pointers to detailed files. Claude reads this automatically at the start of every session. It works alongside `.claude/rules/framework-rules.md`: CLAUDE.md gives context, framework-rules.md gives behavioral instructions.

**Template:**

```markdown
# {Client Name}: {Engagement Description}

## Quick Context

- **Client:** {Client Name}
- **Engagement:** {One-line description}
- **Consulting firm:** {Firm name}
- **Timeline:** {Start date} to {Target end date}
- **Current phase:** {Discovery / Design / Build / Test / Deploy}

## Key Files

| File | What It Contains |
|------|-----------------|
| `engagement/engagement-overview.md` | Full client context, org landscape, stakeholders |
| `engagement/requirements.md` | Business requirements with traceability |
| `engagement/execution-plan.md` | Phase ordering, dependencies, Linear references |
| `engagement/roadmap.md` | High-level timeline, milestones, percent complete |
| `engagement/architecture/solution-design.md` | Cross-epic solution architecture |
| `.framework/config.yaml` | Project metadata and epic registry |

## Epics

| Epic | Prefix | Phase | Key Contact |
|------|--------|-------|-------------|
| {Epic name} | {PREFIX} | {Phase} | {Contact} |

## Rules

Behavioral rules for this project live in `.claude/rules/framework-rules.md`. Read that file for instructions on how to handle questions, decisions, transcript processing, Linear sync, and all other framework operations.
```

#### `.claude/rules/framework-rules.md`
The behavioral rules that govern how Claude Code operates within this project. This is the most important file — it tells Claude what to do when events happen (question raised, question answered, ticket created, etc.). Generated from the template in `claude-md-template.md`.

Note: Salesforce-specific solutioning standards (architecture patterns, coding standards, documentation references) are handled by separate Claude Code skills (e.g., SF Architect, SF Develop) that can be invoked as needed. The framework rules focus only on project management and knowledge tracking behaviors.

#### `.framework/config.yaml`
Project metadata. Claude reads this to understand basic engagement context.

**Template:**

```yaml
client: {Client Name}
engagement: {Engagement Description}
consulting_firm: {Your Firm}
start_date: YYYY-MM-DD
target_end_date: YYYY-MM-DD
current_phase: Discovery
primary_contact:
  name: {Name}
  role: {Role}
  notes: {Any relevant context about this person}
epics:
  - id: {epic-slug}
    name: {Epic Display Name}
    prefix: {2-4 letter prefix for question IDs}
  - id: {epic-slug}
    name: {Epic Display Name}
    prefix: {2-4 letter prefix}
```

```markdown
{List the key standard and custom objects and what they represent in this client's context}

{If there was a previous consulting partner, what did they do? What is the state of their work? Any other relevant history.}
```

### Architecture Level

#### `engagement/architecture/solution-design.md`
The high-level solution architecture for the entire engagement. Describes how the epics relate to each other, what the end-state looks like, and the overall approach. This generates the epics; the technical designs live inside them.

**Template:**

```markdown
# Solution Design — {Client Name}

**Last updated:** YYYY-MM-DD
**Phase:** {Discovery / Design / Build}

## Current State

### Org Architecture
{High-level description of the current Salesforce org — what exists, how it's structured, key data flows}

### Pain Points & Gaps
{What's broken, missing, or inefficient that this engagement is solving}

## Target State

### End-State Architecture
{What the org looks like when this engagement is complete. The "after" picture.}

### Key Changes from Current State
{Summary of the major changes — new objects, modified processes, new integrations, retired components}

## Epic Relationships

{How the epics relate to and depend on each other. Which ones must happen first, which can run in parallel, and why.}

### Data Flow
{How data moves through the system in the target state — between objects, between Salesforce and external systems, between business processes}

### Integration Points
{All external systems that touch Salesforce, what data moves, and in which direction}

## Assumptions
{Key assumptions the design is built on. If any of these prove false, the design needs revisiting.}

## Open Design Questions
{High-level questions that affect the overall architecture, not just a single epic. These should also be tracked in engagement/questions.md.}
```

#### `epics/{epic-name}/questions.md`
Questions scoped to this epic. Same format as engagement-level questions.

**Template:** Same as `engagement/questions.md` but scoped header reads `# Questions — {Epic Name}`

#### `epics/{epic-name}/decisions.md`
Decisions scoped to this epic. Same format as engagement-level decisions.

**Template:** Same as `engagement/decisions.md` but scoped header reads `# Decisions — {Epic Name}`

#### `epics/{epic-name}/technical-design.md`
The technical approach for this epic. A living document that evolves from rough direction in discovery to full specification in design phase.

For simple epics, this is the only design doc. For complex epics, this becomes the high-level design, with detailed designs in the `features/` subdirectory.

**Template:**

```markdown
# Technical Design — {Epic Name}

**Last updated:** YYYY-MM-DD
**Phase:** {Discovery / Design / Build}
**Status:** {Draft / In Review / Approved}

## Overview
{What this design covers and the high-level approach}

## Current State
{Relevant current state specific to this epic — existing objects, processes, data}

## Proposed Approach
{The technical approach. How we're going to build this.}

## Data Model Changes
{New objects, new fields, modified relationships, deleted fields}

## Process & Automation
{Flows, triggers, batch jobs, scheduled jobs — what runs, when, why}

## Migration / Data Considerations
{Data mapping, transformation rules, volume considerations, rollback strategy}

## Testing Approach
{How we validate this works — test scenarios, acceptance criteria at the epic level}

## Open Questions
{Questions specific to this design that need answers before it can be finalized. These should also be tracked in the epic's questions.md.}
```

**Template:**

### Feature Level (Optional)

Features use the same templates as the epic level for `technical-design.md`, `user-stories.md`, and `questions.md`, just scoped to the feature.

Claude creates a `features/` subdirectory only when:
- The epic has identifiable distinct functional areas that each need their own design
- Multiple user stories would benefit from being grouped under a named feature
- The technical design for the epic is complex enough that a single design doc becomes unwieldy

If in doubt, don't create it. You can always restructure later.

### References

#### `references/`
Unprocessed inputs: meeting transcripts, brain dumps, pasted notes. Claude processes these and extracts items into the appropriate framework locations. The raw files are kept for reference but are not the source of truth; the framework files are.

Naming convention: `{date}-{description}.md` (e.g., `2025-01-15-kickoff-transcript.md`)

## Conventions

### Epic Directory Naming
Use kebab-case slugs: `data-migration`, `user-auth-redesign`. These should match the `id` field in `.framework/config.yaml`.

### Engagement-Level Scaffolding
When Claude scaffolds a new engagement, it creates:
1. `CLAUDE.md` at project root
2. `.claude/rules/framework-rules.md`
3. `.framework/config.yaml`
4. `engagement/engagement-overview.md`
5. `engagement/questions.md`
6. `engagement/decisions.md`
7. `engagement/risks.md`
8. `engagement/execution-plan.md`
9. `engagement/requirements.md`
10. `engagement/roadmap.md`
11. `engagement/session-log.md`
12. `engagement/change-requests.md`
13. `engagement/meeting-notes/` (empty directory)
14. `engagement/status-reports/` (empty directory)
15. `engagement/deployment-checklists/` (empty directory)
16. `engagement/architecture/solution-design.md`
17. `references/` (empty directory)

### Epic-Level Scaffolding
When Claude scaffolds a new epic, it creates:
1. `overview.md` (always)
2. `questions.md` (always)
3. `decisions.md` (always)
4. `technical-design.md` (always, even if empty initially)
5. `user-stories.md` (always, even if empty initially)
6. `features/` (only when needed, and only after the epic-level technical design has enough clarity to identify distinct features)

### Cross-References Between Files
Use relative markdown links. Example from a decision in an epic's `decisions.md`:
```markdown
**Driven by:** [Q-DM-003](./questions.md#q-dm-003) — "How do we handle conflicting field values during merge?"
**Impacts:** [Technical Design — Merge Rules](./technical-design.md#merge-rules)
```

### Solution Design vs. Technical Design
- **Solution design** lives at `engagement/architecture/solution-design.md` — it's the big picture across all epics
- **Technical designs** live inside each epic (or feature) — they're the implementation-level detail
- The solution design generates epics; technical designs generate user stories
- If a technical design decision affects the overall architecture, update the solution design too
